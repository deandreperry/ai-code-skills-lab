# Design System Health Audit

## Skill Name

Design System Health Audit

## Description

Assess design system health across adoption, component quality, token consistency, accessibility, documentation, testing, and governance.

## When To Use

- A design system exists but adoption or trust is uneven.
- Teams duplicate components or bypass shared primitives.
- Leadership needs a maturity assessment and investment roadmap.
- A design system is preparing for a major release or platform expansion.

## Inputs Required

- Component library source, docs, token files, usage data, and consumer feedback.
- Release process, contribution model, ownership, and support channels.
- Known bugs, adoption blockers, and cross-team dependencies.
- Target platforms and product areas.

## Step-by-Step Workflow

1. Review coverage, adoption, quality, accessibility, theming, documentation, tests, release process, and governance.
2. Compare component APIs and token layers for consistency and duplication.
3. Identify trust gaps such as missing states, unstable APIs, weak docs, or poor support.
4. Prioritize improvements by product impact, team effort, and adoption leverage.
5. Produce a roadmap that balances foundations, component gaps, and enablement work.

## Output Format

- Design system health scorecard with evidence and priorities.
- Risk register for accessibility, API stability, documentation, and adoption.
- Roadmap with immediate, near-term, and strategic investments.

## Success Criteria

- The audit explains why teams do or do not adopt the system.
- Recommendations balance technical quality with operating model improvements.
- The roadmap is credible for both engineers and design leadership.
- Findings are actionable without requiring a full restart.

## Common Mistakes

- Measuring health only by component count.
- Ignoring contribution experience and support burden.
- Treating design consistency as separate from accessibility and code quality.
- Recommending governance without practical tooling or documentation.

## Example Usage

Use this skill before planning a design system quarter to assess component quality, adoption gaps, token maturity, and documentation health.
