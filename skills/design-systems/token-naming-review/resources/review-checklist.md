# Token Naming Review Review Checklist

Use this checklist while executing the skill.

## Preparation

- [ ] Confirm current token files, naming conventions, and theme examples.
- [ ] Confirm product domains, brand modes, supported platforms, and component library scope.
- [ ] Confirm examples of confusing, duplicated, or disputed token names.
- [ ] Confirm consumer expectations for CSS, JSON, TypeScript, Figma, or native platforms.

## Execution

- [ ] Classify tokens into primitive, semantic, component, and state layers.
- [ ] Identify names tied too tightly to values, themes, implementation details, or one component.
- [ ] Evaluate hierarchy, ordering, prefixes, state names, and discoverability.
- [ ] Propose naming rules with examples and migration mapping.
- [ ] Call out tokens that need design decisions rather than mechanical renaming.

## Completion

- [ ] Names communicate intent across themes and platforms.
- [ ] The structure makes future tokens predictable.
- [ ] Renames are prioritized by impact and migration cost.
- [ ] Design decisions are separated from syntax cleanup.
