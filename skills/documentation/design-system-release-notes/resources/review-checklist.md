# Design System Release Notes Review Checklist

Use this checklist while executing the skill.

## Preparation

- [ ] Confirm commit range, PR list, changelog draft, package version, or diff summary.
- [ ] Confirm audience such as designers, frontend engineers, QA, or product teams.
- [ ] Confirm component, token, theme, documentation, and migration changes.
- [ ] Confirm breaking changes, deprecations, known issues, and upgrade steps.

## Execution

- [ ] Group changes by components, tokens, themes, docs, accessibility, fixes, and breaking changes.
- [ ] Translate implementation details into consumer impact and migration actions.
- [ ] Call out visual changes, API changes, token renames, and required code updates.
- [ ] Include examples or before/after notes when they reduce ambiguity.
- [ ] Flag unclear changes that need maintainer confirmation.

## Completion

- [ ] Consumers understand what changed and whether they need to act.
- [ ] Breaking changes and migration steps are explicit.
- [ ] Design and engineering impacts are both represented.
- [ ] Release notes are accurate and not overloaded with internal noise.
